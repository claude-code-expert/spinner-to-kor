# /tdd-refactor — 마일스톤 구조 개선 (Tidy First)

Kent Beck TDD의 **REFACTOR** 단계다. **마일스톤의 모든 테스트가 통과할 때만** 구조를 바꾼다. 행위는 절대 바꾸지 않는다.

## 전제 확인

1. `TDD-IMPLEMENTS.md`에서 대상 마일스톤을 정한다(`$ARGUMENTS` 또는 상태 `GREEN`인 마일스톤).
2. `npm test`로 **현재 전부 GREEN**임을 먼저 확인한다.
   - GREEN이 아니면 **즉시 중단**하고 "리팩터링 불가: 먼저 /tdd-green"이라 알린다.

## 절차

3. 이번 마일스톤에서 추가/변경된 코드의 정리 대상을 찾는다:
   - **하드코딩된 값**(매직 넘버/문자열) → 이름 있는 상수/인자로.
   - **중복** → 추출(Extract Function)로 제거.
   - 의도가 불명확한 이름 → 의미 있는 이름으로 (Rename).
   - 단일 책임 위반 → 분리. 부작용/상태 최소화.
4. **한 번에 하나의 리팩터링**만 한다. 정립된 패턴은 그 이름으로 부른다(Extract Function, Rename, Inline 등).
5. **각 리팩터링 단계마다 `npm test`를 실행**해 모든 테스트가 여전히 GREEN인지 확인한다. 깨지면 즉시 되돌린다.
6. 끝나면 `TDD-IMPLEMENTS.md`에서 마일스톤 상태를 `DONE`으로 갱신한다.

## 금지

- **새 행위를 추가하지 않는다.** 새 테스트도 추가하지 않는다(그건 다음 마일스톤의 RED).
- 구조적 변경과 행위적 변경을 한 단계/한 커밋에 섞지 않는다.

## 커밋 안내

- 구조적 변경은 **별도 커밋**으로 분리한다. 메시지에 구조적 변경임을 명시한다.
  예: `refactor(parser): 헤더 키를 상수로 추출 (구조적 변경)`
- 작고 잦은 커밋을 사용한다. (커밋은 사용자가 지시할 때만 수행한다)

## 끝

"DONE: <마일스톤 ID> — 적용한 리팩터링: <패턴명 목록>, 테스트 pass N / fail 0" 한 줄로 보고한다.
